
#### Milestone 45.1: Enrich `resource_management` with collected node kinds

<!-- milestone-meta
id: "45.1"
status: "planned"
-->

**Scope:** Populate the existing `resource_management` category for Python, Go, and
Java by classifying node kinds the adapters **already collect and currently drop**:
Python `with_statement` (context managers), Go `defer_statement` (deferred
cleanup), Java `try_with_resources_statement`. Today `resource_management` is
Rust-macro-only; these three languages produce zero instances despite the parsing
stage emitting the hints. Pure node-kind classification — no parsing-layer change,
no new category.

**Why this milestone exists:** Context managers (`with open(...)`), `defer
cleanup()`, and try-with-resources are the canonical resource-acquisition idioms in
their languages and are exactly what `resource_management` is meant to measure. The
hints already exist; only the classifier mapping is missing. Highest value-to-cost
ratio in the cross-language track.

**Deliverables:**
- Extend `resource_management::NODE_KINDS` in
  `crates/sdivi-patterns/src/queries/resource_management.rs` to include
  `with_statement`, `defer_statement`, `try_with_resources_statement` (additive to
  the existing `macro_invocation`).
- Confirm `category_for_node_kind`/`classify_hint` route these correctly (the
  `macro_invocation` special-case in `classify_hint` stays as-is; the new kinds fall
  through to `category_for_node_kind`).
- Update `docs/pattern-categories.md` Python/Go/Java tables (replace the
  "(none in v0)" cells).

**Detection:**
| Language | Node kind | Examples |
|---|---|---|
| Python | `with_statement` | `with open(p) as f:`, `with lock:` |
| Go | `defer_statement` | `defer f.Close()`, `defer mu.Unlock()` |
| Java | `try_with_resources_statement` | `try (var r = open()) { ... }` |
| Rust | `macro_invocation` (unchanged) | `vec!`, `drop!`, `assert!` |

**Migration Impact:** Additive node kinds within an existing category — no
`list_categories()` count change. Python/Go/Java repos gain non-zero
`resource_management` buckets on first post-upgrade snapshot (previously dropped
hints). This shifts per-category counts for those languages — a `MIGRATION_NOTES.md`
event under the existing `resource_management` entry. `snapshot_version` stays `"1.0"`.

**Files to modify:**
- **Modify:** `crates/sdivi-patterns/src/queries/resource_management.rs` — `NODE_KINDS`.
- **Modify:** `docs/pattern-categories.md`, `MIGRATION_NOTES.md`, `CHANGELOG.md`.
- **Verify only:** Python/Go/Java adapters already collect these kinds.

**Acceptance criteria:**
- `category_for_node_kind("with_statement", "python") == Some("resource_management")`.
- A Python fixture with `with` blocks and a Go fixture with `defer` produce instances.
- `category_contract.rs`, clippy/fmt/doc gates green; Rust macro behavior unchanged.

**Tests:**
- Unit: each new node kind classifies; Rust `macro_invocation` logging-split unchanged.
- Integration: Python/Go/Java fixtures count correctly.

**Watch For:**
- **Do not disturb the Rust `macro_invocation` → logging split (M33).** The
  `excludes_callee` logic in `classify_hint` is orthogonal to the new node kinds and
  must stay byte-identical.
- **Cross-language semantic note.** `with`/`defer`/`try-with-resources` are
  conceptually equivalent (scoped resource release) but structurally different; the
  shared category is intentional. Document so cross-language comparison is understood.

**Seeds Forward:**
- Python `with_statement` could be split from explicit `__enter__`/`contextmanager`
  definitions for finer signal; defer until requested.
