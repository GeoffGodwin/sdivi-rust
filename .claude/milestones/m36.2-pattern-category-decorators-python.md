
#### Milestone 36.2: Pattern Category ŌĆö `decorators` (Python)

<!-- milestone-meta
id: "36.2"

**Scope:** Extend the `decorators` category (introduced in M36.1) to Python.
Python decorators cover `@dataclass`, `@property`, `@staticmethod`,
`@app.route(...)` / `@router.get(...)` (Flask/FastAPI), `@pytest.fixture`,
`@cached_property`, `@app.task` (Celery). **This is a free win:** the Python
adapter already collects `decorated_definition` (confirmed in
`crates/sdivi-lang-python/src/extract.rs`) ŌĆö it is currently extracted and dropped
because no category consumes it.

**Why this milestone exists:** Python frameworks (FastAPI, Flask, pytest, Django,
Celery, stdlib dataclasses) express most structural intent through decorators.
Routing this already-collected node kind into `decorators` adds Python coverage at
near-zero cost and keeps the category cross-language and uniform.

**Deliverables:**
- Add `"decorated_definition"` to `decorators::NODE_KINDS` in
  `crates/sdivi-patterns/src/queries/decorators.rs` (created in M36.1).
- Confirm the Python adapter already emits `decorated_definition` hints (it does);
  no adapter change expected. If finer granularity is wanted, the inner `decorator`
  child kind can be added instead ŌĆö see Watch For.
- Update `docs/pattern-categories.md` Python node-kind table.

**Detection:**
| Language | Node kind | Examples |
|---|---|---|
| Python | `decorated_definition` | `@dataclass\nclass C:`, `@app.get("/u")\ndef h():`, `@pytest.fixture\ndef db():` |

**Migration Impact:** Additive within the existing `decorators` category ŌĆö no new
category, so `list_categories()` count is unchanged from M36.1. Python repos gain
a non-zero `decorators` bucket on first post-upgrade snapshot; previously these
hints were dropped. `snapshot_version` stays `"1.0"`. Note in `MIGRATION_NOTES.md`
under the same `decorators` entry.

**Files to modify:**
- **Modify:** `crates/sdivi-patterns/src/queries/decorators.rs` ŌĆö add Python node kind.
- **Modify:** `docs/pattern-categories.md` (Python table), `MIGRATION_NOTES.md`,
  `CHANGELOG.md`.
- **Verify only:** `crates/sdivi-lang-python/src/extract.rs` already lists
  `decorated_definition` ŌĆö no edit expected.

**Acceptance criteria:**
- `category_for_node_kind("decorated_definition", "python") == Some("decorators")`.
- A FastAPI/pytest-shaped Python fixture yields `decorators` instances.
- `category_contract.rs`, clippy/fmt/doc gates green.

**Tests:**
- Unit: `decorated_definition` ŌåÆ `decorators`.
- Integration: Python fixture with `@dataclass` + `@app.get` produces the expected count.

**Watch For:**
- **`decorated_definition` is the wrapper node**, not the `@`-line itself ŌĆö one
  decorated function/class with three stacked decorators is **one**
  `decorated_definition`. If per-decorator granularity is desired (count each `@`),
  switch to the inner `decorator` child kind and add it to the Python adapter's
  `PATTERN_KINDS`. For v0, wrapper-granularity (one per decorated def) is the
  documented, simpler choice ŌĆö state it explicitly in the docs so the count
  semantics are unambiguous and stable.
- **Cross-language count asymmetry.** TS counts each `decorator`; Python (v0) counts
  each `decorated_definition`. Document this difference ŌĆö it is an intentional v0
  simplification, not a bug, and matters for cross-language comparison.

**Seeds Forward:**
- Aligning TS (per-decorator) and Python (per-decorated-def) granularity would
  require the Python adapter to emit inner `decorator` nodes. Defer until a
  cross-language comparison consumer needs symmetric counts.
