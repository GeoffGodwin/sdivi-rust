
#### Milestone 49.1: Leiden Hang — Capture the Minimal Repro and Contain it in CI

<!-- milestone-meta
id: "49.1"
status: "pending"
-->

**Scope:** Convert the silent, intermittent `run_leiden` hang into a fast,
deterministic, reproducible failure that CI surfaces in seconds instead of a
30-minute runner timeout — and capture the exact minimal `(graph, seed)` input
that triggers it, which M49.2 needs to verify its fix. This phase does **not**
change the Leiden algorithm; it makes the existing defect observable and
guarded. After this phase, a pathological case fails the test with its input
printed rather than hanging the job, and a committed regression pins the case.

**Why this milestone exists:** `prop_refine_modularity_does_not_decrease`
(`crates/sdivi-detection/tests/refinement.rs`) generates random inputs (`n in
3..=8`, `raw_edges 1..16`, `seed in 0..=255`) with no fixed seed. For most
inputs `run_leiden` converges in microseconds, but for certain `(graph, seed)`
combinations it does not, and the test hangs. Because `proptest` is pulled
without its `fork`/`timeout` cargo features, `PROPTEST_TIMEOUT` is ignored, so a
non-converging case hangs the whole test binary until the 30-minute CI job
timeout kills the runner — the run then shows as `cancelled` (observed on the
`v0.2.47` and `v0.2.48` `main` CI runs; passed on PR runs only by luck of the
random seed). The hang reproduces locally (e.g. `PROPTEST_CASES=8192 cargo test
-p sdivi-detection --test refinement prop_refine_modularity_does_not_decrease`
runs past a 200s wall cap). Root cause is documented in M49.2; this phase makes
the failure cheap and concrete.

**Deliverables:**

- Enable `proptest`'s `fork` and `timeout` features for `sdivi-detection`'s
  dev-dependency (`proptest = { workspace = true, features = ["fork", "timeout"] }`
  or equivalent at the workspace dep), and set a per-case timeout in the
  `proptest!` block touching `run_leiden` (e.g. `ProptestConfig { timeout: 10_000,
  fork: true, ..with_cases(256) }`). Effect: a non-converging case is killed and
  reported as a test failure **with its minimized input**, in seconds — not a hang.
- Run the now-failing proptest, let it minimize, and **commit the captured input**
  to `crates/sdivi-detection/tests/proptest-regressions/` (proptest persists it).
- Add a deterministic, non-proptest regression `#[test]` in `refinement.rs` (or a
  new `tests/leiden_termination.rs`) that constructs the captured minimal graph +
  seed, runs `run_leiden`, and asserts it returns — guarded so it cannot itself
  hang CI (spawn on a thread with a generous bounded join, or assert via a
  work-counter once M49.2 introduces one). This test is expected to FAIL/timeout
  until M49.2 lands; mark it `#[ignore = "fails until M49.2 fixes leiden blowup; see milestone"]`
  with the seed/graph recorded in a comment so M49.2 un-ignores it.

**Migration Impact:** None. Test-only and dev-dependency-only change. No
`snapshot_version`, schema, config, or public-API change. The `fork` feature
forks test processes on Unix; confirm it is gated/skipped appropriately on
Windows runners (proptest falls back gracefully — verify the matrix stays green
on `windows-latest`).

**Files to create or modify:**

- `Cargo.toml` (workspace `proptest` dep features) and/or `crates/sdivi-detection/Cargo.toml`.
- `crates/sdivi-detection/tests/refinement.rs` — `ProptestConfig` timeout/fork; ignored regression test.
- `crates/sdivi-detection/tests/proptest-regressions/refinement.txt` — committed captured case.

**Acceptance criteria:**

- With the offending input present, `cargo test -p sdivi-detection --test refinement`
  fails (or reports the ignored regression) in **under ~30s**, never hanging.
- The captured minimal `(n, edges, seed)` is recorded in a committed regression file
  and in a code comment, ready for M49.2.
- The full `cargo test --workspace` CI matrix completes (red on the new guard is
  acceptable for this phase; it must not hang) on ubuntu/macos/windows.

**Tests:** The proptest itself (now bounded) plus the new ignored deterministic
regression test are the deliverable. No behavioural assertions change.

**Watch For:** Do not "fix" the hang by shrinking the generator bounds or pinning
a benign seed — that hides the defect M49.2 must fix. The goal is to make the
real pathological case fail FAST and stay captured, not to make it disappear.
Confirm `fork`/`timeout` actually take effect (without the cargo features they
are silently ignored — that is the current bug).

**Seeds Forward:** Hands M49.2 a concrete minimal reproducing input and a guard
test to un-ignore once the algorithm is fixed.
