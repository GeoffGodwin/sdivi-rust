
#### Milestone 45.2: Enrich `error_handling` with collected node kinds

<!-- milestone-meta
id: "45.2"
status: "planned"
-->

**Scope:** Strengthen the existing `error_handling` category for Python and Java by
classifying already-collected, currently-dropped node kinds: Python `except_clause`,
Java `catch_clause` and `throw_statement`. Today `error_handling` counts
`try_statement`/`try_expression`/`match_expression`; the per-clause and throw-site
hints the adapters emit are discarded. Pure node-kind classification — no parsing
change, no new category.

**Why this milestone exists:** A single `try` with five `except`/`catch` arms is
structurally very different from one with a single arm, and `throw`/`raise` density
is a real error-flow signal. Counting only the `try` umbrella loses that variance.
The clause/throw hints already exist; only the mapping is missing.

**Deliverables:**
- Extend `error_handling::NODE_KINDS` in
  `crates/sdivi-patterns/src/queries/error_handling.rs` to include `except_clause`
  (Python), `catch_clause` and `throw_statement` (Java), additive to the existing kinds.
- Update `docs/pattern-categories.md` Python/Java tables.

**Detection:**
| Language | Node kinds | Examples |
|---|---|---|
| Python | `try_statement` (existing), `except_clause` | `except ValueError:`, `except (A, B) as e:` |
| Java | `catch_clause`, `throw_statement` | `catch (IOException e)`, `throw new X()` |
| Rust | `try_expression`, `match_expression` (unchanged) | `x?`, `match r { ... }` |

**Migration Impact:** Additive node kinds within an existing category — no
`list_categories()` count change. Python/Java `error_handling` counts rise on first
post-upgrade snapshot (each clause/throw now counts, not just the `try`).
`MIGRATION_NOTES.md` event under the existing `error_handling` entry.
`snapshot_version` stays `"1.0"`.

**Files to modify:**
- **Modify:** `crates/sdivi-patterns/src/queries/error_handling.rs` — `NODE_KINDS`.
- **Modify:** `docs/pattern-categories.md`, `MIGRATION_NOTES.md`, `CHANGELOG.md`.
- **Verify only:** Python adapter collects `except_clause`; Java adapter collects
  `catch_clause`/`throw_statement` (confirmed in their `extract.rs`).

**Acceptance criteria:**
- `category_for_node_kind("except_clause", "python") == Some("error_handling")`.
- A Python fixture with multi-arm `try/except` counts each `except`.
- `category_contract.rs`, clippy/fmt/doc gates green.

**Tests:**
- Unit: each new node kind classifies.
- Integration: multi-arm Python `try/except` and Java `catch`/`throw` fixtures.

**Watch For:**
- **Double-count semantics.** A Python `try` with 3 `except` arms now yields 1
  `try_statement` + 3 `except_clause` = 4 instances. This is intentional (more arms =
  more structure = higher entropy), but it must be **documented** so the count change
  is understood as designed, not as a regression.
- **Keep `match_expression` Rust-only mapping intact.**

**Seeds Forward:**
- Python `raise` and JS `throw` (the TS/JS adapters do not currently collect
  `throw_statement`) could be added for symmetry via a small adapter change. Defer.
