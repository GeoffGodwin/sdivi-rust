
#### Milestone 28: Leiden Performance — Induced-Subgraph Refinement and Recursion Compression Cap

<!-- milestone-meta
id: "28"
status: "in_progress"
-->

**Scope:** Eliminate two performance bugs in `sdivi-detection`'s Leiden implementation that cause `Pipeline::snapshot` to hang for minutes-to-hours on large, sparse, weakly-connected graphs (e.g. an MFE shell with ~1778 files and ~700 import edges). The first bug forces every `refine_community` call to allocate state for the entire graph and walk every edge, regardless of how few nodes are actually in the coarse community being refined. The second bug only stops the recursive Leiden descent when refinement compresses the graph by literally zero nodes, so a graph that compresses by 1-3% per level recurses dozens of times — each level paying the full per-coarse-community refinement setup cost. After this milestone, refinement is `O(|members| + induced_edges)` per coarse community, recursion stops as soon as compression falls below a configurable ratio (default 10%), and a configurable hard recursion-depth cap (default 32) provides belt-and-suspenders against any pathological case the ratio gate misses. Modularity quality stays inside the existing `verify-leiden` 1% tolerance band; snapshot determinism is preserved. The two new knobs ship as `[boundaries]` config keys (`leiden_min_compression_ratio`, `leiden_max_recursion_depth`) and as fields on `LeidenConfig` / `LeidenConfigInput` so WASM consumers can tune them too.

**Why this milestone exists:** A real adopter (the consumer app's MFE-shell sample) hit a `sdivi snapshot` invocation that ran for over an hour on a 1778-node graph before being killed. Profiling pinpointed `crates/sdivi-detection/src/leiden/refine.rs::refine_community` as the dominant cost. Two compounding bugs were identified:

- **Bug 1 — refinement state is sized to the full graph.** `refine_community` (refine.rs:185-266) allocates `vec![false; n]` (line 197), `(0..n).collect()` (line 203), and calls `RefinementState::from_partition(graph, &singleton_partition, n)` (line 204). `from_partition` (lines 46-76) iterates over every node and every edge in the **entire** graph to populate `sigma_tot`, `inner_edges`, and `size`, even though the move loop afterward only touches `members` (line 212). On a graph that produces hundreds of small coarse communities, this setup cost runs hundreds of times per outer-loop iteration — in practice it dominates the runtime of refinement by 2-3 orders of magnitude on sparse graphs.

- **Bug 2 — recursion only halts on identity compression.** `leiden_recursive` (mod.rs:113-157) breaks the outer loop only when `agg_graph.n >= graph.n` (line 138) — i.e., refinement merged literally zero nodes. On a sparse, weakly-connected graph a single round of refinement typically merges a handful of singletons (1778 → 1750 → 1730 → …) so the identity break never trips, but neither does the algorithm make meaningful structural progress. Each recursion level still pays the full per-coarse-community refinement setup cost from Bug 1, which is what turns a slow snapshot into a hang.

The two bugs interact: Bug 1 makes each level expensive; Bug 2 makes the level count enormous. Either fix alone reduces total work by a constant or a factor; both fixes together make the runtime proportional to the actual structural complexity of the graph rather than to a worst-case product of "outer loop × recursion depth × full-graph refinement setup."

The fix is purely an algorithmic / bookkeeping refactor — the published `LeidenPartition` JSON shape is unchanged, the `verify-leiden` quality gate stays the same (modularity within 1% of leidenalg, community count within ±10%), and snapshot determinism is preserved provided the induced-subgraph ID remap is built deterministically (sorted ascending by global ID, see Watch For).

**Theoretical basis:** Traag et al. 2019, *From Louvain to Leiden*, §2.2 Algorithm 2 (Refinement). The algorithm operates **inside** each coarse community independently — a node `v` in coarse community `S` can only consider moves to sub-communities `C ⊂ S`, and sub-community statistics (`Σ_tot`, `inner_edges`, `size`) only need to range over `S`. The current implementation respects this in the move loop but not in the state setup. Building an induced subgraph on `members` and running refinement on that subgraph is mathematically identical to the current restricted move loop on the full graph, modulo node-ID remapping. The only behavioural difference is performance.

The 10% compression cutoff is not from Traag — Traag-Leiden iterates until convergence, defined as "the local move phase produces no further moves." We retain that primary convergence criterion (the existing `if !moved { break; }` at mod.rs:127 is unchanged) and add a *secondary* cutoff: stop recursing when the aggregated graph would only compress by less than `leiden_min_compression_ratio`, on the grounds that on extremely sparse graphs the algorithm has effectively converged even though a strict identity-compression break would not yet trip. The 1% `verify-leiden` modularity tolerance absorbs the small-but-nonzero quality cost of this early exit on the verify-leiden fixtures (which compress by >50% per level — well above the 10% threshold and therefore unaffected). The recursion-depth cap is a pure safety net for pathological inputs and is set far above any depth a healthy graph would reach (`log₂(1e6) ≈ 20 < 32`).

**Deliverables:**

- **`LeidenGraph::induced_subgraph` (new method on `crates/sdivi-detection/src/leiden/graph.rs`):**
  - Signature: `pub fn induced_subgraph(&self, nodes: &[usize]) -> (LeidenGraph, Vec<usize>)`. Returns the induced subgraph and a `local_to_global: Vec<usize>` map of length `nodes.len()` such that `local_to_global[local_id]` is the original graph's node ID.
  - **Determinism contract:** internally the function takes the input `nodes` slice, copies it into a `Vec`, sorts ascending (`sort_unstable`), and dedups. Local IDs are assigned in that sorted order (smallest global ID gets local ID 0). This is critical for preserving the BTreeMap-iteration tie-break behaviour documented in M18 (`docs/determinism.md` and `refine.rs::refine_community`'s argmax loop).
  - Walks `members` once, building a `BTreeMap<usize, usize>` (`global_id → local_id`) from the sorted-and-deduped list. Walks each member's `adj` list once; for each neighbour, looks up the local ID in the map (skip if not present — the edge crosses out of `members`). Self-loops on a member propagate to the local node's `self_loops` slot.
  - Edge weights and self-loops are preserved exactly. Cross-community edges are dropped (Traag refinement only considers intra-coarse-community moves; cross-edges are out of scope for refinement at this level).
  - Returns a `LeidenGraph` of `n = members_sorted_dedup.len()`. The standard `from_edges_weighted` invariants hold — adjacency sorted, weights parallel, `degree` includes 2× self-loop, `total_weight = cross_total + self_loop_total`.
  - Add doc test demonstrating: a 5-node K₅ graph, induce on `[0,2,4]`, assert resulting graph is K₃ with the right edge weights and `local_to_global == [0,2,4]`.

- **Rewrite `refine_community` (`crates/sdivi-detection/src/leiden/refine.rs:185`):**
  - Build `(sub, local_to_global) = graph.induced_subgraph(members)`.
  - Construct `RefinementState::from_partition(&sub, &(0..sub.n).collect::<Vec<_>>(), sub.n)` — singleton init, but on the **sub** graph. State arrays are now sized to `members.len()`, not `n`. Setup cost drops from `O(n + total_edges)` to `O(|members| + induced_edges)`.
  - The shuffle order is built from `(0..sub.n).collect()` (local IDs) rather than `members.to_vec()` (global IDs). Determinism is preserved because the sorted-ascending local-ID assignment in `induced_subgraph` produces the same iteration order *under the rng-shuffled permutation* as today (the shuffle is over indices, not over IDs — same length, same seed, same permutation index → same shuffled positions; the *content* at each shuffled position is the same node either way).
  - The neighbour walk inside the move loop iterates `sub.adj[node]` instead of `graph.adj[node]`. The `in_coarse: Vec<bool>` filter (refine.rs:197-200) is no longer needed — the subgraph definitionally contains only `members`. Delete the `in_coarse` allocation and its filter check.
  - The argmax `BTreeMap<usize, f64>` accumulator now keys on local sub-community IDs. After the move loop converges, write back `state.assignment[local]` into `refined[local_to_global[local]]` for every `local` in `0..sub.n`.
  - **Critical determinism note (carry into Watch For):** the global `refined: &mut [usize]` buffer has globally unique singleton-init IDs (set by `refine_partition` at line 171). Writing local sub-community IDs into it would collide between sibling `refine_community` calls. Two ways to avoid the collision:
    1. After the move loop, collapse `state.assignment` into globally unique IDs by adding a per-call offset chosen so no other coarse community can overlap (e.g., `members[0]` — the smallest member ID — works because singleton-init at the outer level guarantees `members[0]` itself is a unique ID and we have at most `|members|` distinct sub-communities).
    2. Use a `sub_id → global_id` lookup that maps each distinct sub-community to a unique global ID drawn from `members` (e.g., the smallest member of each sub-community).
  - Pick option (2) — it's mechanically simpler and matches the structure of `renumber_in_place` already in use. Document the choice inline.

- **`leiden_min_compression_ratio` and `leiden_max_recursion_depth` config keys (`crates/sdivi-config/src/config.rs::BoundariesConfig`):**
  - Add `pub leiden_min_compression_ratio: f64` to `BoundariesConfig`. Default `0.1`. Doc: "Stop the recursive Leiden aggregation when a recursion level would compress the graph by less than this fraction (e.g. `0.1` = stop when fewer than 10% of nodes were merged). Lower values produce slightly higher modularity at the cost of much longer runtime on sparse, weakly-connected graphs. Default `0.1` is empirically safe across the verify-leiden fixtures (which compress by >50% per level)."

A graph healthy enough to reach this depth has structure log₂(n)-deep at minimum — `32` corresponds to ~4 billion nodes, well above any realistic codebase. Exists as a belt-and-suspenders safeguard against pathological inputs that the compression-ratio cutoff fails to catch."

  - Validate at config load: `leiden_min_compression_ratio` must be in `[0.0, 1.0]` (exclusive on `1.0`; a value of `1.0` would terminate at every level). `leiden_max_recursion_depth` must be `≥ 1`. Out-of-range values produce `ConfigError::InvalidValue { key, message }` (exit 2).

- **`leiden_recursive` consumes the new knobs (`crates/sdivi-detection/src/leiden/mod.rs:113`):**
  - Change signature to `fn leiden_recursive(graph, initial, rng, max_iter, quality, gamma, min_compression_ratio: f64, max_recursion_depth: u32, depth: u32)`. The `depth` parameter starts at `0` for the top-level call and increments by 1 on each recursive call.
  - At the top of the function, return `partition` (the renumbered initial assignment, no recursion) when `depth >= max_recursion_depth`. Emit a `tracing::warn!` once per top-level invocation if the cap is hit (track via a `depth_capped: bool` flag passed back; or simpler, log unconditionally at the deepest call site — pick whichever stays cleaner).
  - Replace the `if agg_graph.n >= graph.n { break; }` early-exit with: `if (agg_graph.n as u64) * 10 > (graph.n as u64) * 9 { break; }` — integer arithmetic for `min_compression_ratio = 0.1`. **However**, since `min_compression_ratio` is a runtime parameter, not a compile-time constant, the comparison must use it. Two options:
    1. `let threshold = (graph.n as f64) * (1.0 - min_compression_ratio); if (agg_graph.n as f64) > threshold { break; }` — float comparison, deterministic but introduces a float into the hot path.
    2. Compute `let threshold_n = ((graph.n as u64) * ((1000.0 * (1.0 - min_compression_ratio)).round() as u64)) / 1000;` then compare integer-to-integer. Quantises the ratio to 0.001 resolution which is far finer than anyone would tune.
  - Pick option (1) — the float comparison is on a single derived `threshold`, which is computed once per recursion level (not per inner-loop iteration) and is deterministic given a fixed `min_compression_ratio`. The KDD-determinism story holds. Add a comment citing this choice.
  - Forward the new params to the recursive call: `leiden_recursive(&agg_graph, agg_init, rng, max_iter, quality, gamma, min_compression_ratio, max_recursion_depth, depth + 1)`.
  - Update both `run_leiden` (mod.rs:46) and `run_leiden_with_weights` (mod.rs:82) call sites to source the two new params from `cfg`. The `depth` argument is `0` at this top level.

**Watch For:**

- **Determinism: induced-subgraph ID remap MUST sort members ascending.** If `induced_subgraph` is implemented to keep members in input order, the BTreeMap argmax tie-break in `refine_community` would produce a different (deterministic, but different) partition than M18. The verify-leiden 1% band would likely still absorb this, but the existing `proptest_seeded.rs` "bit-identical across 100 runs" test gate would still pass (same seed → same shuffle → same outcome) while the *content* of the partition silently shifts. Be explicit: sort `members` ascending inside `induced_subgraph`, document the contract on the function, and add a test case asserting `local_to_global` is monotonically increasing.
- **Sub-community ID collision after writeback.** `refine_partition` (the outer caller) uses globally unique singleton IDs (`(0..graph.n).collect()`) and relies on `refine_community`'s output not colliding across sibling calls. The new induced-subgraph implementation produces local IDs `[0, |members|)` per call — these MUST be remapped to globally unique IDs before writeback, or `renumber_in_place` at the end of `refine_partition` will incorrectly merge sub-communities from disjoint coarse communities. Use the "smallest member of each sub-community" mapping documented in Deliverables. Add a test: two disjoint coarse communities, refinement produces sub-communities, no global ID is shared between them.
- **`#[serde(default)]` for the two new `LeidenConfigInput` fields.** A WASM consumer with a hand-rolled JSON `LeidenConfigInput` must keep deserialising without error after this milestone. Always use `#[serde(default)]` (or `#[serde(default = "fn_name")]`) on additions. The same applies to `BoundariesConfig` (TOML-parsed). Test by deserializing a fixture JSON / TOML that omits both new fields; assert the defaults take effect.
- **Float comparison in the early-exit.** The chosen implementation does `let threshold = graph.n as f64 * (1.0 - min_compression_ratio); if agg_graph.n as f64 > threshold { break; }`. This is deterministic given a fixed `min_compression_ratio` because the float ops happen on the same compiler / target / FMA settings as everywhere else in the pipeline. Document inline that this ONE call site is the only float comparison influencing partition shape. If a future cross-platform-bit-identity push happens (still flagged as "Unresolved" in CLAUDE.md), re-evaluate.
- **`tracing::warn` at recursion-cap boundary.** Rule 11: warnings go to stderr. Make sure the `target` is `sdivi_detection::leiden` (or some sdivi-prefixed target), not the bare module path, so consumers can filter. Don't accidentally emit to stdout — running `sdivi show --format json | jq '.'` must remain clean (the existing `tests/stdout_stderr_split.rs` test will catch a regression here).
- **`renumber_in_place` interaction.** The end-of-`refine_partition` `renumber_in_place(&mut refined)` is what gives the final compact `[0, k)` ID space. If the per-call writeback in `refine_community` uses non-contiguous IDs (e.g., gaps because a sub-community ended up empty), `renumber_in_place` handles it — no change needed. Just verify with a test that an empty mid-range sub-community doesn't break ID assignment.
- **`map_to_aggregate_init` and `flatten_partition` (`aggregate.rs:79, 94`)** are unchanged by this milestone but they consume `refined` and `partition` arrays that might now contain different IDs than M18. Verify they still work — no logic change needed, but read them carefully and add an inline comment in `aggregate.rs` explaining that they are agnostic to the specific IDs used.
- **Doc-comment placement when adding new fields.** Per CLAUDE.md "Doc comment placement when inserting items": every new `pub` field in a struct gets its own `///` block, separated from the next field's `///` block by either a blank line or the field declaration itself (the `,` is sufficient). `#![deny(missing_docs)]` on `sdivi-core` will fire if a new `pub` field on `LeidenConfigInput` lacks a doc comment.
- **Performance regression on dense graphs.** The induced-subgraph approach pays a small upfront cost (allocating `local_to_global`, building the `BTreeMap<usize, usize>`) per `refine_community` call. On a graph with very FEW, very LARGE coarse communities (e.g., all nodes in one community), the allocation cost might exceed the savings. Run the verify-leiden `large` fixture's wall-clock pre-M28 vs post-M28 — the build-out and the allocation should net out positive even there because verify-leiden's large is sparse-ish. If not, optimize: use a `Vec<u32>` global-to-local map sized to `graph.n` and reused across refine_community calls (but then we re-introduce an `O(n)` per-call setup, defeating the point). The right answer is probably "BTreeMap is fine; profile only if a fixture regresses."
- **WASM `.d.ts` regeneration.** When you change `WasmLeidenConfigInput`, the generated `.d.ts` must be regenerated. Verify the M24 distribution job (npm publish) still works locally with `wasm-pack build`. The release pipeline (M13) ties this to its publish step.
- **`detect_boundaries` callers in tests.** `crates/sdivi-core/tests/` and `bindings/sdivi-wasm/tests/` may construct `LeidenConfigInput` directly; the two new fields default-deserialize when missing, but a hand-constructed `LeidenConfigInput { ... }` literal will fail to compile until updated. Audit the call sites with `rg "LeidenConfigInput\s*\{" crates bindings tests examples` and either use `LeidenConfigInput::default()` with `..default()` field-init shorthand or update the literals.
- **Cargo.lock churn.** This milestone shouldn't add or remove any dependency. If `Cargo.lock` changes outside the workspace's own crates, investigate before committing.
- **Probe scaffolding cleanup.** If you write any `_probe.rs`-style ad-hoc test files during debugging, delete them before commit (CLAUDE.md hygiene; M18 had the same warning).

**Seeds Forward:**

- **Per-coarse-community parallelism.** The induced-subgraph approach makes `refine_community` calls genuinely independent per coarse community. A future milestone could `rayon::par_iter` over `community_members` in `refine_partition` for a multi-core speedup. Determinism would require per-community RNGs seeded by `seed XOR community_id` (or similar), which is a non-trivial change to the seeding contract — defer until a real adopter needs it. The KDD-5 "no CSR view" decision still holds; this is a different axis of optimization.
- **Adaptive compression ratio.** Static `0.1` may be wrong for some graph shapes. A future milestone could compute the ratio adaptively from level 0's compression — e.g., "stop when the compression ratio drops below 50% of the level-0 ratio." Adds complexity; leave as `Seeds Forward` until a user complains about either over- or under-aggressive cutoff.
- **Recursion budget instead of depth cap.** A more sophisticated alternative to `max_recursion_depth` is a wall-clock budget threaded through the call stack (`Instant`-based) that aborts gracefully if the algorithm would exceed N seconds. Useful for CI-as-gate scenarios but introduces a non-deterministic exit path — defer.
- **Profile-guided refinement skip.** For very small coarse communities (e.g., `members.len() <= 2`), refinement is mathematically a no-op (a 2-node community is already singleton-or-merged, no sub-structure possible). The current code already filters `members.len() <= 1`; a small follow-up could lift the threshold to 2 with a proof. Tiny perf win; not worth a milestone on its own but worth bundling into the next perf pass.
- **Documenting the modularity drift in `docs/determinism.md`.** This milestone introduces a documented case where snapshot modularity may shift across a code-version boundary by up to ~1%. Add a section to `docs/determinism.md` titled "Modularity stability across sdivi-rust versions" that calls out M18 and M28 as the two known cutovers, and explains why the verify-leiden 1% band is the contract rather than bit-identity.
- **`leiden_quality.rs` perf assertion.** The verify-leiden suite asserts modularity quality but not runtime. Adding a soft wall-clock budget (e.g., `large` fixture must complete in <300s in debug) would catch perf regressions earlier. Defer — premature without a perf-CI plan.
