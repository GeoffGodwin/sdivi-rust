
#### Milestone 50: Disable `wasm-opt` post-processing — binaryen-free WASM builds (v0.2.51)

<!-- milestone-meta
id: "50"
status: "pending"
-->

**Scope:** Make `bindings/sdivi-wasm` build cleanly with `wasm-pack` in
network-restricted / corporate environments that lack the `binaryen` toolchain
(`wasm-opt`), and ship the change as v0.2.51. By default `wasm-pack build`
runs `wasm-opt` as a post-processing pass and, when the binary isn't on PATH,
tries to download it — which fails behind a corporate proxy/airgap, breaking the
build for the project's primary audience. The size-optimised `[profile.release-wasm]`
Cargo profile (`opt-level = "z"`, fat LTO, `codegen-units = 1`) already delivers the
primary size reduction, so the marginal extra shave from `wasm-opt` is not worth the
hard toolchain dependency. This milestone disables `wasm-opt` via wasm-pack package
metadata, verifies the binaryen-free build is valid and within the existing size
budget, documents the change, and bumps the workspace to `0.2.51`. No source/behaviour
change to the analysis surface; `snapshot_version` stays `"1.0"`.

**Why this milestone exists:** Operator feedback after the v0.2.50 release: the
primary audience runs in corporate environments where `binaryen`/`wasm-opt` is
frequently unavailable, so any consumer (or CI mirror) building `sdivi-wasm` from
source hits a `wasm-pack` failure trying to obtain `wasm-opt`. Measured locally
(wasm-pack 0.13.1): with `wasm-opt` disabled the bundler `.wasm` is **1,679,707 bytes**
vs **1,531,325 bytes** with `wasm-opt` — a ~148 KB (~10%) increase that stays under the
existing **1,835,008-byte (1.75 MB)** per-target budget with ~155 KB of headroom. So the
trade-off (drop a fragile toolchain dependency for ~10% size) is favourable and needs no
budget change.

**Deliverables:**

1. **Disable `wasm-opt` in wasm-pack metadata.** In `bindings/sdivi-wasm/Cargo.toml`,
   immediately **before** the final comment line (`# Release profile settings are set
   at workspace root in Cargo.toml (profile.release-wasm).`), add:

   ```toml
   [package.metadata.wasm-pack.profile.release]
   wasm-opt = false
   ```

2. **Rebuild + verify locally (no commit of `pkg/`).** `pkg/` is gitignored and
   regenerated by `build.sh` / `release.yml`; **do not** commit `pkg/` artifacts.
   Run `wasm-pack build --target bundler --release` (and `--target nodejs`) from
   `bindings/sdivi-wasm/` with the `release-wasm` settings and confirm: (a) the build
   succeeds **without invoking `wasm-opt`** (no binaryen download/run in the log),
   and (b) the resulting `pkg/bundler/sdivi_wasm_bg.wasm` is within `WASM_BUDGET_BYTES`
   (≈1.68 MB < 1.75 MB). The published npm artifact is rebuilt by `release.yml` on the
   tag and will reflect the new profile automatically.

3. **CHANGELOG `[0.2.51]` entry** under a new dated section:
   > **Changed — Disabled `wasm-opt` post-processing in the wasm-pack release profile.**
   > Binaryen (`wasm-opt`) is often unavailable in network-restricted build
   > environments; the `release-wasm` Cargo profile (`opt-level = "z"`, fat LTO)
   > already provides the primary size reduction. The bundler/node `.wasm` grows
   > ~10% (≈1.53 MB → ≈1.68 MB) but stays within the 1.75 MB per-target budget.
   > `snapshot_version` stays `"1.0"`.

4. **Version → 0.2.51.** The workspace `[workspace.package].version` becomes `0.2.51`,
   and **`bindings/sdivi-wasm/pkg-template/package.json` must be synced to `0.2.51`**
   in the same change (the `workspace_version` test asserts they match, and the Tekhton
   auto-bump does NOT update the JSON file — see Watch For).

**Migration Impact:** None to schema, config, public API, or analysis output.
`snapshot_version` stays `"1.0"`. The npm `@geoffgodwin/sdivi-wasm` `.wasm` is ~10%
larger than v0.2.50 but functionally identical (same wasm-bindgen surface). No new
config keys, no new `pub` items, no `MIGRATION_NOTES.md` entry required.

**Files to create or modify:**

- `bindings/sdivi-wasm/Cargo.toml` — add the `[package.metadata.wasm-pack.profile.release]` block.
- `CHANGELOG.md` — `[0.2.51]` section.
- `Cargo.toml` (`[workspace.package].version` → `0.2.51`) and
  `bindings/sdivi-wasm/pkg-template/package.json` (`"version": "0.2.51"`); `Cargo.lock` follows.
- (No `pkg/` files — gitignored, CI-regenerated.)

**Acceptance criteria:**

- `wasm-pack build --target bundler --release` (and `--target nodejs`) in
  `bindings/sdivi-wasm/` succeed **without** running or fetching `wasm-opt`.
- The built bundler/node `.wasm` is ≤ `WASM_BUDGET_BYTES` (1,835,008); the `wasm.yml`
  `Check bundle sizes` gate stays green.
- `cargo test --workspace` green, including `wasm_package_json_version_matches_workspace`
  (Cargo `0.2.51` == pkg-template `0.2.51`).
- `cargo clippy -- -D warnings`, `cargo fmt --check`, and `cargo doc` (`-D warnings`) clean.
- CHANGELOG has a `[0.2.51]` section with the wasm-opt note.

**Tests:** No new test files. The existing `wasm.yml` size gate and the
`workspace_version` test are the regression guards. Verification is the local
binaryen-free `wasm-pack build` plus the green CI matrix on the release commit.

**Watch For:**

- **Version double-bump.** If run under Tekhton, the finalize stage auto-bumps the
  patch version. Do **not** also hand-edit `Cargo.toml` to `0.2.51` in the coder stage
  or the result lands at `0.2.52`. Let the auto-bump set Cargo.toml, and ensure
  `pkg-template/package.json` ends at the SAME version (the auto-bump misses it — this is
  the exact drift that blocked M49; tekhton m43 tracks the tool-side fix). If running
  manually, bump both files together.
- **Do not commit `pkg/`.** It is gitignored on purpose (`bindings/sdivi-wasm/.gitignore`).
  Step 2 is local verification only.
- Confirm the `wasm-opt = false` key sits under `[package.metadata.wasm-pack.profile.release]`
  exactly (wasm-pack silently ignores a misspelled table and would keep running `wasm-opt`).

**Seeds Forward:** If a future consumer needs the extra `wasm-opt` shave, it can be
re-enabled in a CI-only build (where binaryen IS available) without reintroducing the
hard dependency for source builders — e.g. a `release.yml`-only `wasm-opt -Oz` step.
