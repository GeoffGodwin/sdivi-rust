## Summary
M13 introduces a tag-driven release workflow (`.github/workflows/release.yml`) with manual approval gates for crates.io and npm publishing, along with a thin `sdi-cli` library extraction and two 3-line binary entry points. The Rust code changes (`lib.rs`, `main.rs` stubs) contain no security-relevant logic. The workflow is structurally sound — manual environment gates guard both publish jobs, `cargo audit` blocks `create-release`, and sensitive tokens are handled via GitHub Secrets. The findings below are hardening gaps in the CI pipeline, not exploitable vulnerabilities in the distributed software.

## Findings
- [MEDIUM] [category:A08] [.github/workflows/release.yml] fixable:yes — All third-party actions are pinned to version tags (`@v2`, `@v4`, `@stable`) rather than immutable SHA digests. A compromised action release or moved tag on `actions/checkout`, `Swatinem/rust-cache`, `softprops/action-gh-release`, or `rustsec/audit-check` could exfiltrate `CARGO_REGISTRY_TOKEN` or read build artifacts before the manual gate is reached. Fix: replace each `uses: owner/action@vN` with `uses: owner/action@<full-sha>  # vN` using the commit SHA for that tag.
- [LOW] [category:A08] [.github/workflows/release.yml:231] fixable:yes — `publish-npm` installs the Rust toolchain via `dtolnay/rust-toolchain@master`, a mutable branch ref. If the `master` branch of that action is updated between workflow runs, arbitrary code could run with access to the WASM build environment. Fix: pin to a released tag (`dtolnay/rust-toolchain@v1`) or a SHA, consistent with the `@stable` ref used in all other jobs.
- [LOW] [category:A05] [.github/workflows/release.yml] fixable:yes — No workflow-level `permissions:` block is declared. Without it, `GITHUB_TOKEN` retains the repository default (which is `write-all` unless the org/repo has tightened it). Adding `permissions: {}` at the workflow root with explicit per-job overrides (e.g. `contents: write` only on `create-release`) enforces least privilege regardless of org settings.
- [LOW] [category:A05] [.github/workflows/release.yml:159-160] fixable:yes — `CARGO_REGISTRY_TOKEN` is exported as a job-level `env:` var in `publish-crates`, making it visible to every step in the job including `actions/checkout@v4` and `Swatinem/rust-cache@v2`. Scoping it to only the `cargo publish` steps via step-level `env:` limits exposure if a third-party action reads environment variables.
