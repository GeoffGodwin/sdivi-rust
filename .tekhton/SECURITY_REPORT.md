## Summary
M16 introduces `commit_extract.rs`, which shells out to `git` and `tar` to resolve commit references and extract historical trees. All git command arguments are passed via Rust's `Command::args()` (no shell interpolation), and the resolved SHA is validated as exactly 40 hexadecimal characters before being used in subsequent commands, which substantially limits exploitation risk. The main residual concerns are a missing `--` separator that could allow git-flag injection for the initial ref resolution step, tar extraction without explicit path-traversal hardening, and verbose error messages that echo git stderr. No secrets, auth bypasses, network calls, or cryptographic misuse were found. The remainder of the change (helpers refactor, error variant, CLI glue) is free of security concerns.

## Findings
- [MEDIUM] [category:A03] [crates/sdi-pipeline/src/commit_extract.rs:40-47] fixable:yes — `reference` is passed directly to `git rev-parse --verify <reference>` without a `--` separator. Rust's `Command::args()` does not invoke a shell so there is no shell injection, but a reference starting with `-` (e.g. `--local-env-vars`, `--show-toplevel`) could be interpreted as a git option. The subsequent 40-hex-char SHA validation prevents meaningful exploitation in practice, but the library surface is public and could be called with externally-sourced references. Fix: change the args slice to `["rev-parse", "--verify", "--", reference]`.
- [LOW] [category:A01] [crates/sdi-pipeline/src/commit_extract.rs:115-120] fixable:yes — `tar -xC <tmpdir>` is invoked without path-traversal mitigations. `git archive` normalises tree paths and disallows `../` components, so exploitation requires a crafted git tree object, but defence-in-depth supports adding `--no-absolute-filenames` (GNU tar). Fix: add `"--no-absolute-filenames"` to the tar `args` call.
- [LOW] [category:A09] [crates/sdi-pipeline/src/commit_extract.rs:13-33] fixable:yes — `CommitExtractError` variants embed raw git and tar stderr verbatim (e.g. `RefResolutionFailed { stderr }`, `ArchiveFailed { stderr }`). For a CLI tool this is acceptable, but if these errors are forwarded to structured logs or external services they may disclose repository layout details. Fix: truncate stderr strings beyond a safe length (e.g. 512 chars) before storing them in the error variants.
