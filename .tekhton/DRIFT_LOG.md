## M28 Leiden Perf Bugs — Discovered and Fixed (2026-05-07)

M28 records two pre-existing perf bugs in the M18-era refinement implementation:

- **Bug 1 — full-graph state allocation per coarse community.** `refine_community` allocated `vec![false; n]` (in_coarse) and called `RefinementState::from_partition(graph, ..., n)` iterating all `n` nodes and all edges at every call, regardless of the coarse community size. On a sparse graph with hundreds of small coarse communities this setup cost ran hundreds of times per outer-loop iteration, dominating runtime by 2–3 orders of magnitude.

- **Bug 2 — recursion only halted on identity compression.** `leiden_recursive` broke the outer loop only when `agg_graph.n >= graph.n` (zero nodes merged). On sparse, weakly-connected graphs a single round of refinement typically merged only a handful of singletons (e.g. 1778 → 1750 → 1730 → …) so the identity break never fired, while neither did the algorithm make meaningful structural progress. Each recursion level still paid the full per-coarse-community setup cost.

Both bugs were correctness-irrelevant for the `verify-leiden` fixtures (small/medium/large all compress >50% per level and have <100 nodes total), so the fixture suite did not catch them. New regression coverage: a `leiden_perf.rs` test on a 1500-node 600-edge sparse graph that asserts wall-clock <5s in debug (passed in ~1.2s).

## Import Resolution Drift Discovered During bifl-tracker Validation (2026-05-04)

- [2026-05-04 | "step 1: stand sdivi up against bifl-tracker"] **Two compounding bugs in the dependency-graph stage produce near-empty graphs on every non-Rust language, silently invalidating every coupling-based metric since M05.** Discovered while running `sdivi snapshot` on `bifl-tracker` (a real Next.js TS app, 38 source files): `nodes=38 edges=0 communities=38 modularity=0.000`. Same symptom previously observed on a large internal TS project at work where `boundary_violations` always reported `0`. Root causes:
  - **Adapter layer:** `extract_imports` in `sdivi-lang-{python,typescript,javascript,go,java}` pushes the entire `import_statement` / `import_declaration` text into `FeatureRecord::imports`, e.g. `"import { foo } from '../lib/x';"`. Only `sdivi-lang-rust` correctly extracts the specifier. The graph stage's `resolve_import` checks `import.starts_with("./") || import.starts_with("../")` plus the Rust prefixes, all of which fail because the string starts with the `import` keyword. Every non-Rust import is dropped at `DEBUG`.
  - **Resolver layer:** even with clean specifiers, `resolve_relative` (sdivi-graph/src/dependency_graph.rs:220) strips `../` characters with `trim_start_matches` but never walks up a directory. `from=components/Bar.tsx`, `import="../lib/foo"` looks for `components/lib/foo.tsx`. Multi-level `../../` strips all dot-prefixes greedily without bound. Additionally, no resolver handling exists for Python dotted imports, Go module paths, Java packages, or tsconfig path aliases — so even with correct specifier strings, those languages would produce zero or near-zero edges.
- **The bug was observed and pinned, not fixed.** Test `relative_import_parent_slash_strips_prefix_resolves_in_same_dir` in `crates/sdivi-graph/tests/dependency_graph.rs:152` explicitly asserts the broken behavior with comment `"See BUG note in TESTER_REPORT.md re: parent navigation not implemented"`. The test was passing the regression suite for ~20 milestones because it pinned the wrong behavior as expected.
- **Why this stayed hidden through M19 (`compute_boundary_violations` implementation):** M19 shipped a real boundary-violation calculator on top of an empty graph. Boundary violations require crossing edges; with zero edges, the function correctly returned zero violations and was indistinguishable from a working implementation. The `verify-leiden` quality suite (M18) uses synthetic graphs (manually built adjacency lists), bypassing the parsing-and-resolution path entirely — so Leiden quality stayed verified while the inputs to Leiden in real-world snapshots were trivial. Same pattern as the M17/M18 Leiden modularity regression: a critical end-to-end test was missing because the unit-test layer pinned a subset of behavior that didn't exercise the broken seam.
- **Remediation (added to MANIFEST 2026-05-04):**
  - **M25:** Adapter Module-Specifier Extraction. Fix all 5 non-Rust adapters to emit specifier-only strings. Rust adapter unchanged. Independent unit-testable change in 5 crates.
  - **M26:** Resolver Parent Navigation + Per-Language Module Conventions. Rewrite `resolve_relative` to navigate `..` properly; introduce per-language extension lists and resolution conventions (Python dotted, Go module-path via `go.mod`, Java dotted with conventional source roots); split `super::` parent-walk out of the Rust path while preserving stem-search fallback. Pluralizes the resolver API to `Vec<NodeIndex>` for Java wildcard / Go package multi-edge cases. Includes explicit Multi-Language Regression Guarantees section ensuring Rust resolution is byte-identical pre- and post-M26 except for the `super::` improvement.
  - **M27:** tsconfig Path Alias Support. JSONC-tolerant parse of `compilerOptions.paths` + `baseUrl`. Layered between relative resolution and external-fall-through in the TS/JS dispatch arm only. Zero impact on Rust/Python/Go/Java.
- **Regression discipline lessons (carry forward):**
  - **Pinning broken behavior in tests is worse than no test.** When a tester observes a bug and the implementer can't fix it that cycle, the test should be `#[ignore = "BUG: ..."]` with a tracking issue, not a passing test that asserts the wrong outcome. The "BUG note" comment is a red flag that should have blocked merge.
  - **End-to-end fixtures per supported language are non-negotiable.** The `verify-leiden` suite and synthetic FeatureRecord tests both sidestepped the parsing-and-resolution path. Going forward, `tests/per_language_baselines.rs` (added in M25/M26) pins exact edge counts on real fixtures for all 6 supported languages — every PR touching parsing, resolution, or graph construction must update this file deliberately or fail.
  - **Coupling-based metrics need a "non-trivial graph" guard.** A snapshot with `edge_count == 0` and `node_count > 1` is almost certainly an instrumentation failure, not a real signal. Add a stderr `WARN` from `Pipeline::snapshot` when the resolver drops more than 90% of imports — would have surfaced this on day one. (Tracked as a Seeds Forward in M26; consider promoting to its own milestone if M25/M26/M27 don't surface it via fixture coverage.)
  - **Multi-language risk is structural, not incidental.** sdivi supports 6 languages and adapters/resolvers must remain symmetric. M25/M26/M27 each include a Multi-Language Regression Guarantees section enumerating which languages are touched and which are byte-identical. Any future resolver/adapter milestone should follow the same pattern.
