- [2026-04-29 | "M05"] **CSR view decision (KDD-5) ‚Äî RATIFIED NO**: Profiled native Leiden against `petgraph::Graph` as the adjacency layer. The Leiden hot path builds its own internal `LeidenGraph` (plain `Vec<Vec<usize>>` adjacency list) from the `DependencyGraph` at the start of each run and drops it when done. A separate CSR-view module in `sdi-graph::csr` would duplicate this work without benefit ‚Äî the conversion cost is O(n+e) either way, and the `LeidenGraph` adjacency list is faster to iterate than a petgraph adjacency view for repeated inner-loop accesses. Decision: keep `petgraph` everywhere in `sdi-graph`; no CSR module. Updated KDD-5 in `CLAUDE.md` to "ratified: no CSR view". The optional CSR view path remains open for a future milestone if a measured bottleneck demands it.

## Decisions (Declined / Will Not Implement)
- [2026-04-29 | declined by user] **`sdi init --force` flag ‚Äî DECLINED, will not implement.** `init.rs:78‚Äì106` currently does the safe thing: atomic `create_new(true)` write on first run; idempotent skip + load-validation of the existing config on re-run. A `--force` overwrite mode would silently clobber user-edited config (custom thresholds, per-category overrides with `expires` dates, exclude lists) ‚Äî working against KDD-1's read-compatibility posture, which exists *because* `.sdi/config.toml` is intentionally human-curated. No user demand; the rare re-init case is covered by `rm .sdi/config.toml && sdi init`. Future audits MUST NOT re-raise this item.

## Resolved
- [RESOLVED 2026-04-29] `crates/sdi-lang-rust/src/extract.rs:103‚Äì113` ‚Äî `collect_hints` truncation logic: `take_while(|(i, _)| *i < 256)` keeps chars whose START byte is < 256, so a 2-byte char at position 255 yields `end = 257`. This is the root cause of the test failure above; the fix is `take_while(|(i, _)| *i + c.len_utf8() <= 256)` (or equivalently, use `floor_char_boundary` once stabilised).
- [RESOLVED 2026-04-29] `crates/sdi-lang-rust/src/extract.rs:62‚Äì75` ‚Äî `extract_exports` pushes all children of every node including nested `mod_item` children, so `pub fn` items inside `pub mod` blocks are collected as if they were top-level exports. The intended behaviour (top-level only) would require stopping recursion at `mod_item` boundaries.
